- `src/App.tsx` — pass route to CaptureScreen

**Behavior:**
1. When a menu parse returns `incomplete: true`, the opening line says **"This wasn't a complete menu."** (one sentence, nothing more) before the normal section list.

### 2. VoiceOver Accessibility Fixes (2 of 3 P0 Issues) ✅

**Source:** VOICEOVER-AUDIT.md (generated by sub-agent; 3 P0, 9 P1, 10 P2 findings)

**Completed:**

#### P0 #2: Recording Phase Announces into Open Mic
**Problem:** When phase transitions to "recording" and aria-live announces it, VoiceOver's voice gets transcribed by the open speech recognizer as the user's utterance, breaking the interaction.

**Fix:** `src/screens/ConversationScreen.tsx` line 407
```tsx
<div
  role="status"
  aria-live={phase === 'recording' ? 'off' : 'polite'}
  aria-label={indicator.label}
  className={`phase-indicator phase-${phaseClass(phase)}`}
>
```
Now aria-live is OFF during recording, preventing VoiceOver from being transcribed.

#### P0 #3: 60 Seconds of Silent Progress (Find + Analysis)
**Problem:** While searching (9-60s) or analyzing photos (5-10s), the reassurance phrases ("Still searching…", "Almost there…") were spoken but never written to the DOM, so VoiceOver users heard nothing and thought the app was frozen.

**Fix:** Both screens now call `setStatus(msg)` alongside `speak(msg)`:
- `src/screens/FindScreen.tsx` line 52-56: changed reassurance interval to call `announce(msg)` instead of `speak(msg)` directly
- `src/screens/CaptureScreen.tsx` line 84-88: added `setStatus(msg)` inside the analysis reassurance interval

Now progress is both spoken AND visible in the `role="status"` region.

#### P0 #1: App TTS Talks Over VoiceOver (NOT FIXED)
**Problem:** Every screen (LoginScreen, FindScreen, UrlScreen, SavedScreen, ConversationScreen) calls `speak()` at mount, which fires simultaneously with `Screen` auto-focusing `<main>`. VoiceOver announces the screen at the same moment the app TTS begins, creating a cacophony. No global app-voice toggle exists (only ConversationScreen has one).

**Status:** Requires a profile-level `appVoice: boolean` setting gating all `speak()` calls in `src/lib/speech.ts`. **Left for next session.**

### REVIEW.md
**Generator:** Sub-agent `a9d4ea8f114bfcbf6`  
**Findings:** 1 critical, 6 major, 9 minor  
**Location:** C:\Users\2fire\All Coding\menuvoice2\REVIEW.md

**Critical (must fix):**
1. **SSRF in `api/_menuCore.ts:110` (`fetchOne`)**  
   Server fetches user-supplied (menu-from-url) and LLM/prompt-injectable (find-menu stage 2) URLs with `redirect: 'follow'` and **zero private-IP/localhost/metadata-endpoint protection**. Attacker can reach internal services or AWS metadata. Plus full response is parsed/returned to client.  
   **Fix:** Reject URLs matching `/(localhost|127\.|192\.168\.|10\.|172\.1[6-9]\.|169\.254|169\.255|\[::|0\.)/i` before fetch.

**Major (high priority):**
1. **Telemetry batch wedging (`src/lib/telemetry.ts:88`)**  
   `fetch(..., {keepalive: true})` rejects bodies >64 KiB. Failed batch is unshifted and retried forever. Single oversized/poison batch permanently wedges telemetry, grows `_queue` unbounded, re-serializes to localStorage on every `track()`.  
   **Fix:** Check batch size before fetch; if >64 KiB, drop oldest events or split into sub-batches.

2. **FindScreen re-entrancy (`src/screens/FindScreen.tsx:53, 84`)**  
   No re-entrancy guard. User hits Enter, `find()` runs. While loading, user hits Enter again (or race on network slowdown). `loading` closure is stale; both calls proceed. First 9-second reassurance interval orphans and speaks "Still searching…" forever, cutting off the menu being read aloud after navigation. **Session-breaking for a blind user at the table.**  
   **Fix:** Add `useRef` guard: if `find()` already in flight, return early.

3. **Unvalidated LLM categories crash client (`api/find-menu.ts:100-104`)**  
   `result.categories` is cast to array but never validated. If LLM returns garbage (e.g. object instead of array of items), client TypeError becomes the error message, which is then spoken aloud.  
   **Fix:** Validate categories structure before use; throw FriendlyError if invalid.

4. **Coach utterances drop on iOS Safari (`src/lib/speech.ts`)**  
   `cancel()` synchronously stops speechSynthesis, then immediately calls `speak()`. iOS Safari silently drops the new utterance (not queued). App's primary platform, primary UX (guidance during capture) affected.  
   **Fix:** Async/promise-based queue or check `speechSynthesis.paused` before speak.

5. Two more major issues documented in REVIEW.md.
